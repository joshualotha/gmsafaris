## 2026-05-07 — Fix 404 Images on Admin Safari/Destination Edit Pages

**What was done:**
1. **Diagnosed root cause** — The admin edit pages for safaris and destinations used `Storage::url()` to render existing images, but seed data stores paths with `img/` prefix (e.g., `img/serengeti-header.jpg`). These files live in the `public/img/` directory and must be served via `asset()` or the existing `resolveImageUrl()` helper methods. `Storage::url()` generates URLs like `/storage/img/serengeti-header.jpg` which maps to `storage/app/public/img/` — a directory that doesn't exist.

2. **Fixed `resources/views/admin/destinations/edit.blade.php`** — Replaced 3 `Storage::url()` calls with `Destination::resolveImageUrl()`:
   - Line 109: `Storage::url($destination->hero_image)` → `Destination::resolveImageUrl($destination->hero_image)`
   - Line 119: `Storage::url($destination->thumbnail_image)` → `Destination::resolveImageUrl($destination->thumbnail_image)`
   - Line 133: `Storage::url($img)` → `Destination::resolveImageUrl($img)` (gallery loop)

3. **Fixed `resources/views/admin/safaris/edit.blade.php`** — Replaced 1 `Storage::url()` call with `Safari::resolveImageUrl()`:
   - Line 178: `Storage::url($imgPath)` → `Safari::resolveImageUrl($imgPath)` (gallery loop)
   - Note: Hero/thumbnail images in this view already used `Safari::resolveImageUrl()` correctly — only the gallery section was broken.

4. **Cleaned up diagnostic artifacts** — Removed the `/_diag/images` diagnostic route from `routes/web.php` (lines 122-142). The diagnostic logging previously added to `Safari.php` and `Destination.php` models was already reverted.

**What changed:**
- `resources/views/admin/destinations/edit.blade.php` — All 3 image previews now use `Destination::resolveImageUrl()` which handles both `img/` (public directory) and `destinations/` (storage) path prefixes
- `resources/views/admin/safaris/edit.blade.php` — Gallery image previews now use `Safari::resolveImageUrl()` which handles both `img/` and `safaris/` path prefixes
- `routes/web.php` — Removed the diagnostic `/_diag/images` route

**Expected impact:**
- Admin safari edit page — gallery images stored with `img/` prefix (from seed data) will now render correctly instead of showing 404s
- Admin destination edit page — hero, thumbnail, and gallery images stored with `img/` prefix will now render correctly
- Admin-uploaded images (stored as `safaris/...`, `destinations/...`) continue to work via `Storage::url()` fallback in the `resolveImageUrl()` methods
- No impact on public-facing pages — they already used `resolveImageUrl()` correctly
- The `content-script.js` browser extension error is unrelated to the site and requires no action
